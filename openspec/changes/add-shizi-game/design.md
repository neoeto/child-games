## Context

`child-games` 是多游戏纯网页仓库，已有 `wuziqi/`（JoJo 五子棋）作为首个游戏并确立了项目约定：纯 vanilla HTML/CSS/JS、零运行时依赖、零构建工具、PWA、平板触控优先、家长控制 baby-gate、Web Audio API 合成音效、localStorage 命名空间持久化（`<游戏名>:<字段>`）。大厅 `index.html` 有 3 个 placeholder 卡片槽位等待新游戏接入。

本变更新增第二个游戏 `shizi/`（识字乐园），面向 3-4 岁幼儿园小班孩子。核心差异于既有游戏：这是**学习类**游戏（非对战类），需要全新的进度跟踪、内容数据、发音系统，同时复用既有的触控/家长控制/音效约定。

设计基于 4 项调研：①仓库现状与 wuziqi 架构模式 ②HanziWriter 笔顺库（v3.7.3, MIT, 36KB, 0 依赖, vanilla JS 友好）③Web Speech API 浏览器兼容性 ④Make Me a Hanzi 笔画数据源（Arphic PL，9000+ 字覆盖）。

## Goals / Non-Goals

**Goals:**

- 3-4 岁孩子能独立完成「听音选字」玩法（无需家长全程陪同）
- 家长能在「亲子认字」模式中引导孩子认字，通过笔顺动画辅助讲解
- 完整学习闭环：认字 → 听音辨字 → 错字复习 → 巩固掌握
- 进度持久化与正反馈机制（星星/关卡/贴纸）驱动持续学习
- 完全离线运行（PWA 安装后无需网络），HanziWriter 与笔画数据全部本地化
- 保持项目核心原则：纯 vanilla、零构建、零运行时网络依赖

**Non-Goals:**

- ❌ 不做看图识字（避免图片资源依赖，字+音为中心）
- ❌ 不做用户账号、云存档、跨设备同步（localStorage 单设备本地玩）
- ❌ 不做语音识别（孩子发音评判由家长主观判定，保护自信心）
- ❌ 不做手写练习模式（HanziWriter `quiz()` 笔画描写功能留待 V2）
- ❌ 不做 i18n 框架（zh-CN 硬编码，对齐既有约定）
- ❌ 不做多音字完整支持（仅 `pinyin` 字段标注，TTS 用默认读音）
- ❌ 不预制发音音频文件（用系统原生 speechSynthesis，避免 100 字 × mp3 的工作量与体积）
- ❌ 不做 Service Worker 离线缓存（依赖浏览器 HTTP 缓存 + PWA manifest，离线增强留待 V2）

## Decisions

### 决策 1：技术栈 — 克隆 wuziqi 模式，纯 vanilla JS 多文件

**选择**：`shizi/` 目录下 ~10 个文件，结构对齐 `wuziqi/`，每个 `.js` 是 IIFE + window 全局 + `typeof` 守卫跨模块调用。

**理由**：
- 与既有游戏约定完全一致，降低维护成本
- 纯网页双击即玩，对家长分享极友好
- 零构建工具、零 npm 依赖，符合项目原则
- IIFE + window 全局模式已被 wuziqi 验证可行

**备选方案**：
- ❌ 引入 ES modules + 构建工具 — 违反项目零构建原则
- ❌ 抽公共框架（shared/audio.js 跨游戏复用）— 目前仅 2 个游戏，过早抽象；保持每游戏自包含

### 决策 2：双模式架构 — 听音选字（孩子独立）+ 亲子认字（家长引导）

**选择**：
- **听音选字模式**：speechSynthesis 播放发音 → 显示 4 个汉字大字卡 → 孩子点选 → 答对庆祝/答错鼓励重试（错项变灰不退出）→「再听一次」按钮常驻
- **亲子认字模式**：显示巨大字卡（200px+ 字号）→ 笔顺自动播放 → 家长考孩子读音 → 点字听标准发音 → 家长判定「✅会了」(字+1星) / 「🌱再练」(进错字本)
- 主界面提供两个模式入口卡片 + 「我的字库」进度入口

**理由**：
- 双模式覆盖「孩子自己玩」和「家长陪玩」两种场景
- 听音选字锻炼听觉辨识（输出方），亲子认字锻炼发音表达（输入方），互补
- 家长主观判定而非语音识别 — 3-4 岁发音不准，机器评判会挫败感强；家长判定可鼓励为主
- 「答错不惩罚」保护 3-4 岁自信心，符合学前教育正向反馈原则

**备选方案**：
- ❌ 看图识字 — 用户明确否决，图片资源依赖重
- ❌ 单一模式 — 覆盖场景单一，玩法深度不足
- ❌ 语音识别判发音 — 技术不可靠（幼儿发音不标准），UX 灾难

### 决策 3：笔顺播放 — HanziWriter 本地化部署，不走 CDN

**选择**：
- 将 `hanzi-writer@3.7.3` 的 `dist/hanzi-writer.min.js`（36KB）下载到 `shizi/vendor/hanzi-writer.min.js`
- 100 字的笔画 JSON 从 `hanzi-writer-data@2.0.1` 批量下载到 `shizi/data/chars/<字>.json`
- 用自定义 `charDataLoader` 从本地 `fetch('data/chars/<字>.json')` 加载，替代默认 CDN
- 认字模式下集成：显示字卡时自动 `animateCharacter()` → 动画完毕 `then()` 链接 `speechSynthesis.speak()`

**理由**：
- 本地化保持项目「零运行时网络依赖」原则（PWA 离线可用）
- HanziWriter v3.7.3 MIT 协议、0 运行时依赖、vanilla JS 单标签接入，完美契合
- 笔画数据 Arphic PL 允许未修改使用，商业安全
- 100 字笔画数据总计约 150KB（每字 1-3KB），体积可控
- `animateCharacter()` 返回 Promise，便于链式衔接发音

**备选方案**：
- ❌ CDN 引入 HanziWriter — 违反用户明确要求「不要 CDN」
- ❌ 自研 SVG 路径动画（用 makemeahanzi 原始数据）— 工作量 ~3.5 天，重复造轮子，效果不如成熟库
- ❌ 静态笔画分解图 — 无动画，无法体现书写过程，效果弱

### 决策 4：发音方案 — Web Speech API (speechSynthesis)，零素材

**选择**：
- 用浏览器原生 `window.speechSynthesis.speak(new SpeechSynthesisUtterance(char))` 朗读汉字
- 启动时检测 `speechSynthesis.getVoices()` 是否有 zh-CN voice，无则降级显示 pinyin 作视觉提示
- 家长设置中提供语速调节（默认 0.8，范围 0.5-1.5，小孩需要慢速）
- 首次用户交互时触发首次发音（iOS Safari 需用户手势激活）

**理由**：
- 零素材成本（不需要 100 个 mp3）
- 零网络依赖（系统原生）
- iOS Safari / Chrome Android 中文 TTS 支持都不错
- 语速可调对幼儿关键 — 默认语速对孩子太快

**备选方案**：
- ❌ 预制 MP3 音频包 — 100 字 × mp3 = 大量素材工作 + 体积膨胀，违反「不做音效外部资源文件」原则精神
- ❌ 混合方案（关键字预制 mp3 兜底）— V2 增强，本期先用纯 TTS 验证

### 决策 5：进度持久化 — 单 key 打包进度 + 设置类独立 key

**选择**：
- `shizi:progress`：整个进度 JSON（unlockedLevel / words字字典 / errorBook / stickers / streak / stats）打包成一个字符串
- `shizi:soundEnabled`：音效开关（对齐 wuziqi:soundEnabled）
- `shizi:speechRate`：TTS 语速（0.5-1.5）
- `shizi:strokeSpeed`：笔顺动画速度（0.5-2.0）
- 进度 JSON 含 `version: 1` 字段，未来 schema 变更可平滑迁移

**理由**：
- 100 字进度如果每字一个 key 会很碎；打包成单 key 读写 O(1)，总数据约 10KB（远低于 5MB 上限）
- 设置类（音效/语速/笔顺速度）独立 key — 高频读写，避免触发整个 progress JSON 序列化
- `version` 字段是数据迁移的标准做法
- 与 wuziqi 命名空间约定一致（`<游戏名>:<字段>`）

**备选方案**：
- ❌ 每字一个 key（`shizi:word:w001` 等）— 读写碎片化，清空/备份困难
- ❌ IndexedDB — 100 字数据量根本用不上，过度设计

### 决策 6：字库结构 — 100 字 / 10 关 / 主题分组 / JS 常量 + 本地 JSON

**选择**：
- 业务数据存 `shizi/data/words.js`（hardcoded JS 数组，对齐 wuziqi 的 FAMILY_LABELS 模式），每字含 `{id, char, pinyin, level, tip}`
- 笔画数据存 `shizi/data/chars/<字>.json`（100 个独立文件，HanziWriter 格式）
- 10 关主题分组：①数字 ②自然 ③天地方位 ④身体 ⑤家庭 ⑥动作 ⑦颜色大小 ⑧动物 ⑨食物 ⑩常用
- 每关 10 字，按难度递进（数字最易，常用虚词最难）

**理由**：
- 100 字数据 JS 常量化最简单（wuziqi 已有先例），避免 `fetch()` 异步复杂性
- 笔画数据是 HanziWriter 库的输入格式，保持原样不二次加工
- 主题分组符合幼儿认知规律（同类字一起学易记忆）
- 10 字/关卡的粒度适合单次学习时长（3-4 岁专注力 ~10 分钟）

**备选方案**：
- ❌ 全 JSON 化（words.json + fetch）— 100 字 JS 常量够轻量，无需异步
- ❌ 不分组随机 — 幼儿学习需要结构化递进

### 决策 7：星级与解锁机制 — 3 星制 + 60% 门槛

**选择**：
- 每字最多 3 星：
  - ⭐ 亲子认字模式家长点「会了」一次（初步认识）
  - ⭐⭐ 听音选字答对 1 次（听音能辨）
  - ⭐⭐⭐ 听音选字累计答对 3 次（不同场次）（牢固掌握）
- 关卡解锁：上一关 60% 字达 ⭐⭐ 即解锁下一关
- 第 1 关默认解锁

**理由**：
- 3 星制提供渐进式目标，避免「全有全无」挫败感
- 60% 门槛不算苛刻（3-4 岁不会卡关），又能防止跳跃式学习
- 听音选字累计 3 次才满星 — 单次答对可能是猜的，多次验证才牢固
- 家长判定给 ⭐ 保护孩子自信（家长可宽松判定）

**备选方案**：
- ❌ 全部 3 星才解锁 — 过于严格，易卡关
- ❌ 关卡全开放 — 失去结构化递进意义
- ❌ 100% 满星才解锁 — 较真家长向，不适合 3-4 岁

### 决策 8：干扰项策略 — 同关卡随机

**选择**：听音选字的 4 个选项中，正确字 + 同关卡随机抽 3 字作为干扰。

**理由**：
- 同关随机 = 同主题字（如「一二三...」互为干扰），复习效果强
- 3-4 岁分不清近音字（天/田/甜）或形近字（日/曰），近音干扰会变猜谜
- 实现简单，无难度梯度逻辑

**备选方案**：
- ❌ 近音字干扰 — 3-4 岁辨音能力不足，挫败感强
- ❌ 动态难度（掌握后启用近音）— V2 增强，本期保持简单

### 决策 9：正反馈三件套 — 错字本 + 打卡贴纸 + 通关庆典

**选择**：
- **错字本**：听音选字答错的字自动进 `errorBook` 数组，下次听音选字优先从错字本抽（权重 50%）
- **打卡贴纸墙**：连续 7 天玩 = 1 枚贴纸；连续 30 天 = 特殊贴纸；每通关 1 关 = 1 枚贴纸
- **通关庆典**：通关一关时全屏撒花 + JoJo 跳舞动画 + 庆祝音效（约 3 秒）

**理由**：
- 错字本符合艾宾浩斯遗忘曲线 — 错字优先复习
- 打卡贴纸培养每日学习习惯
- 通关庆典给阶段性成就感
- 三者都用 CSS 动画 + Web Audio 合成，零素材

**备选方案**：
- ❌ 复杂的艾式算法（按遗忘曲线精确调度）— 过度工程，3-4 岁学习随机性大
- ❌ 积分商城 — 引入消费主义，不符合幼儿教育理念

### 决策 10：大厅接入 — 替换首个 placeholder

**选择**：根 `index.html` 中第一个 `.game-card.placeholder` 替换为 `.game-card.active` 指向 `shizi/index.html`。

**理由**：
- 最小侵入式接入，不动既有 wuziqi 卡片
- 保留剩余 2 个 placeholder 给未来游戏

## Risks / Trade-offs

- **[风险] iOS Safari speechSynthesis 中文 voice 不可用** → 启动时检测 `getVoices()`，无中文 voice 时 UI 显示 pinyin 作视觉降级提示；README 注明推荐设备。

- **[风险] speechSynthesis 发音机械感** → 这是零素材成本的合理取舍；语速调慢（默认 0.8）+ 重复听可缓解；V2 可为关键字预制 mp3 兜底。

- **[风险] HanziWriter 笔画数据某些字笔画顺序有争议（如「为」「发」）** → 数据源 makemeahanzi 基于中国大陆规范笔顺（GB 13000），与教育部《义务教育语文课程标准》一致；个别争议字 README 注明。

- **[风险] 100 字笔画数据需手动批量下载** → 实现阶段用 curl 脚本一次性下载，写入 README 的「资源更新」章节供未来扩展字库时复用。

- **[风险] 3-4 岁孩子误触返回键退出游戏丢进度** → 每次答对/家长确认/通关立即写 localStorage；游戏内不放大返回按钮（靠系统手势/物理返回）。

- **[权衡] 错字本用简单权重（50%）而非精确遗忘曲线** → 3-4 岁学习随机性大，精确算法过度工程；50% 权重足以提升错字曝光率。

- **[权衡] 无 Service Worker 离线缓存** → PWA manifest 让可安装，浏览器 HTTP 缓存基本够用；完整离线是 V2 增强。

- **[权衡] 字库固定 100 字不可扩展** → 字库是 hardcoded JS 常量，扩展需改代码；但 `data/words.js` 独立文件，修改不影响逻辑代码。V2 可考虑 fetch JSON 化。

## Migration Plan

不适用 — 新增独立 `shizi/` 目录，不影响既有 wuziqi 游戏。仅修改根 `index.html` 替换一个 placeholder 卡片。

回滚策略：根 `index.html` 的卡片替换是唯一侵入点，git revert 即可完全回滚。
